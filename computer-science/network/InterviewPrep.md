## 1. 네트워크 기초 & OSI 계층

### Q. OSI 7계층이란 무엇이고, 왜 계층을 나누었나요?

OSI 7계층은 네트워크 통신을 7단계로 나눈 국제 표준 모델입니다. 1계층 물리 계층부터 2계층 데이터링크, 3계층 네트워크, 4계층 전송, 5계층 세션, 6계층 표현, 7계층 응용 계층 순으로 구성됩니다.

계층을 나누는 이유는 각 계층이 독립적으로 동작하기 때문에 특정 계층이 변경되더라도 다른 계층에 영향을 주지 않고, 문제가 발생했을 때 해당 계층에만 집중해서 디버깅할 수 있습니다. 또한 표준화를 통해 서로 다른 벤더의 장비들이 함께 동작할 수 있습니다.

---

### Q. TCP/IP 4계층과 OSI 7계층의 차이를 설명해주세요.

TCP/IP 4계층은 네트워크 액세스 / 인터넷 / 전송 / 응용 계층으로 구성됩니다. OSI 7계층의 응용 + 표현 + 세션 계층을 응용 계층 하나로 합치고, 데이터링크 + 물리 계층을 네트워크 액세스 계층 하나로 합친 형태입니다.

OSI 7계층은 이론적 표준 모델이고, TCP/IP 4계층은 실제 인터넷에서 사용하는 실용적 모델입니다. 현실에서는 TCP/IP 모델을 기반으로 동작합니다.

---

### Q. 캡슐화(Encapsulation)와 역캡슐화(Decapsulation)란 무엇인가요?

캡슐화는 데이터를 전송할 때 각 계층을 내려가면서 해당 계층의 헤더를 추가하는 과정입니다. 응용 계층에서 시작한 메시지는 전송 계층에서 TCP/UDP 헤더가 붙어 세그먼트가 되고, 네트워크 계층에서 IP 헤더가 붙어 패킷이 되고, 데이터링크 계층에서 MAC 헤더와 트레일러가 붙어 프레임이 됩니다.

역캡슐화는 수신 측에서 반대로 각 계층을 올라가며 헤더를 하나씩 제거하고 데이터를 읽는 과정입니다.

---

### Q. 허브(Hub)와 스위치(Switch)의 차이를 설명해주세요.

허브는 1계층 장비로, 수신한 데이터를 연결된 모든 포트에 브로드캐스트합니다. 모든 장치가 같은 충돌 도메인을 공유하기 때문에 성능과 보안이 낮습니다.

스위치는 2계층 장비로, MAC 주소 테이블을 보고 목적지 포트에만 프레임을 전달합니다. 포트마다 독립적인 충돌 도메인을 가지기 때문에 허브보다 성능과 보안이 훨씬 우수합니다.

---

### Q. MAC 주소와 IP 주소의 차이는 무엇인가요?

MAC 주소는 2계층에서 사용하는 48비트의 물리적 주소로, 네트워크 인터페이스 카드에 하드웨어적으로 고정되어 있습니다. 같은 네트워크 안에서 장치를 식별하는 데 사용합니다.

IP 주소는 3계층에서 사용하는 논리적 주소로, 네트워크 환경에 따라 변할 수 있습니다. 서로 다른 네트워크 간에 통신할 때 사용합니다.

인터넷 통신 시 목적지 네트워크까지는 IP 주소로 라우팅하고, 같은 네트워크 안에서 최종 장치를 특정할 때는 MAC 주소를 사용합니다.

---

### Q. ARP(Address Resolution Protocol)란 무엇인가요?

IP 주소로 MAC 주소를 알아내기 위한 프로토콜입니다.

동작 과정은, 먼저 호스트가 목적지 IP의 MAC 주소를 모르면 ARP 요청을 브로드캐스트로 네트워크 전체에 전송합니다. 그러면 해당 IP를 가진 호스트가 자신의 MAC 주소를 담은 ARP 응답을 유니캐스트로 반환합니다. 요청자는 받은 IP-MAC 매핑 정보를 ARP 캐시에 저장해 이후에 재사용합니다.

---

## 2. IP & 라우팅

### Q. IPv4와 IPv6의 차이를 설명해주세요.

IPv4는 32비트 주소 체계로 약 43억 개의 주소를 표현할 수 있고, 10진수로 표기합니다. 반면 IPv6는 128비트 주소 체계로 사실상 무한한 주소를 표현할 수 있고, 16진수로 표기합니다.

또한 IPv4는 헤더 크기가 가변적이고 IPSec 보안이 선택 사항이며 NAT가 필요합니다. IPv6는 헤더 크기가 40바이트로 고정되고 IPSec이 기본 내장되어 있으며 충분한 주소 공간 덕분에 NAT가 필요 없습니다.

IPv6는 IPv4 주소 고갈 문제를 해결하기 위해 등장했습니다.

---

### Q. 서브넷 마스크와 서브네팅이란 무엇인가요?

서브넷 마스크는 IP 주소에서 네트워크 부분과 호스트 부분을 구분하는 비트 마스크입니다. 예를 들어 `192.168.1.0/24`에서 앞 24비트가 네트워크 주소이고 나머지 8비트가 호스트 주소입니다.

서브네팅은 하나의 큰 네트워크를 여러 개의 작은 서브넷으로 분할하는 것입니다. 브로드캐스트 트래픽을 줄이고 보안을 강화하며 IP 주소를 효율적으로 사용하는 것이 목적입니다.

---

### Q. NAT(Network Address Translation)란 무엇인가요?

사설 IP 주소를 공인 IP 주소로 변환해주는 기술입니다.

하나의 공인 IP로 여러 장치가 인터넷에 접속할 수 있어 IPv4 주소 고갈 문제를 완화합니다. 또한 내부 네트워크 구조를 외부에 숨겨 보안 효과도 있습니다. 우리가 흔히 쓰는 공유기가 대표적인 NAT 장치입니다.

---

### Q. DHCP란 무엇인가요?

Dynamic Host Configuration Protocol로, 네트워크에 연결된 장치에 IP 주소를 자동으로 할당해주는 프로토콜입니다.

동작 과정은 DORA라고 부르는 4단계로 이루어집니다. 먼저 클라이언트가 DHCP 서버를 찾는 Discover 메시지를 브로드캐스트로 전송하면, 서버가 사용 가능한 IP 주소를 제안하는 Offer를 보냅니다. 클라이언트가 해당 IP를 요청하는 Request를 보내면, 서버가 최종 확인하는 ACK를 전송하면서 IP 주소가 할당됩니다.

---

### Q. ICMP란 무엇인가요?

Internet Control Message Protocol로, IP 통신 중 발생하는 오류를 보고하고 네트워크 상태를 진단하기 위한 프로토콜입니다.

`ping` 명령어는 ICMP Echo Request와 Reply를 이용해 대상 호스트가 응답하는지 확인하고, `traceroute` 명령어는 TTL 값을 1씩 증가시키며 목적지까지 경유하는 라우터를 추적합니다.

---

## 3. TCP & UDP

### Q. TCP와 UDP의 차이를 설명해주세요.

TCP는 연결 지향 프로토콜로, 3-way handshake를 통해 연결을 맺은 후 데이터를 전송합니다. 순서 보장, 재전송, 오류 검출, 흐름 제어, 혼잡 제어를 모두 지원해 신뢰성이 높지만 그만큼 오버헤드가 크고 속도가 느립니다. HTTP, 파일 전송, 이메일 같이 데이터 정확성이 중요한 곳에 사용합니다.

UDP는 비연결 지향 프로토콜로, 연결 수립 없이 바로 데이터를 전송합니다. 신뢰성은 보장하지 않지만 헤더가 8바이트로 가볍고 속도가 빠릅니다. 스트리밍, 온라인 게임, VoIP, DNS처럼 속도와 실시간성이 중요한 곳에 사용합니다.

---

### Q. TCP의 3-way handshake를 설명해주세요.

TCP 연결을 수립하는 3단계 과정입니다.

먼저 클라이언트가 서버에 SYN 패킷을 보내 연결을 요청합니다. 서버는 요청을 수락하면서 동시에 자신도 연결을 요청하는 SYN-ACK 패킷을 보냅니다. 마지막으로 클라이언트가 서버의 요청을 수락하는 ACK 패킷을 보내면 연결이 수립됩니다.

---

### Q. TCP의 4-way handshake를 설명해주세요.

TCP 연결을 종료하는 4단계 과정입니다.

먼저 클라이언트가 연결 종료를 요청하는 FIN 패킷을 보냅니다. 서버는 받았다는 ACK를 보내고, 아직 전송할 데이터가 남아있을 수 있으므로 계속 데이터를 보냅니다. 서버가 전송을 모두 마치면 FIN 패킷을 보내고, 클라이언트는 최종 ACK를 보낸 뒤 TIME_WAIT 상태를 거쳐 연결이 종료됩니다.

3-way가 아닌 4-way인 이유는 서버가 FIN을 받은 시점에 아직 보낼 데이터가 남아있을 수 있어, ACK와 FIN을 분리해서 전송하기 때문입니다.

---

### Q. TIME_WAIT 상태란 무엇인가요?

TCP 연결 종료 후 클라이언트가 일정 시간(보통 60~120초) 동안 유지하는 상태입니다.

필요한 이유가 두 가지 있습니다. 첫째, 마지막 ACK가 네트워크에서 소실됐을 때 서버가 FIN을 재전송하면 이에 응답하기 위해서입니다. 둘째, 네트워크에 아직 떠돌고 있는 지연 패킷이 새로운 연결에 섞여 들어오는 것을 방지하기 위해서입니다.

---

### Q. TCP의 흐름 제어(Flow Control)와 혼잡 제어(Congestion Control)의 차이를 설명해주세요.

흐름 제어는 수신자의 처리 속도에 맞게 송신자의 전송 속도를 조절하는 것으로, 수신자 버퍼 오버플로우를 방지하는 게 목적입니다. 슬라이딩 윈도우 방식을 사용해, 수신자가 처리 가능한 버퍼 크기를 Window Size로 알려주면 송신자는 ACK 없이 그 크기만큼 연속해서 전송합니다.

혼잡 제어는 네트워크 자체의 혼잡 상태를 감지해 송신 속도를 줄이는 것으로, 네트워크 전체의 혼잡을 방지하는 게 목적입니다. Slow Start 방식으로 처음에는 천천히 전송하고 ACK를 받을 때마다 윈도우 크기를 2배씩 늘리다가, 혼잡이 감지되면 AIMD 방식으로 윈도우를 절반으로 줄이고 이후 선형으로 증가시킵니다.

---

### Q. UDP를 사용하는 경우는 언제인가요?

신뢰성보다 속도와 실시간성이 중요한 경우에 UDP를 사용합니다.

스트리밍 서비스는 약간의 패킷 손실보다 지연이 더 큰 문제이고, 온라인 게임은 실시간 위치 데이터가 손실돼도 최신 상태만 중요합니다. VoIP는 재전송을 기다리는 것보다 빠른 전달이 중요하고, DNS 조회는 짧은 쿼리와 응답으로 TCP 연결 오버헤드가 불필요합니다. 또한 브로드캐스트나 멀티캐스트처럼 다수에게 동시에 전송할 때도 UDP를 사용합니다.

---

### Q. 소켓(Socket)이란 무엇인가요?

네트워크 통신의 양 끝점(Endpoint)으로, IP 주소와 포트 번호의 조합으로 식별됩니다. 운영체제가 제공하는 API로, 응용 프로그램이 네트워크를 통해 데이터를 주고받을 수 있게 해줍니다.

TCP 소켓은 `bind → listen → accept → connect → send/recv → close` 순서로 동작하고, UDP 소켓은 연결 수립 없이 `bind → sendto/recvfrom → close` 순서로 동작합니다.

---

## 4. DNS

### Q. DNS란 무엇이고, 동작 과정을 설명해주세요.

Domain Name System으로, 사람이 읽기 쉬운 도메인 이름을 IP 주소로 변환하는 분산 데이터베이스 시스템입니다.

조회 과정은 다음과 같습니다. 브라우저 DNS 캐시를 먼저 확인하고, 없으면 OS의 캐시와 hosts 파일을 확인합니다. 거기도 없으면 ISP가 제공하는 로컬 DNS 서버에 질의합니다. 로컬 DNS 서버는 루트 DNS 서버에 물어서 .com 같은 TLD 서버 주소를 받고, TLD 서버에 다시 물어서 해당 도메인의 권한 있는 DNS 서버 주소를 받습니다. 마지막으로 권한 있는 DNS 서버에서 실제 IP 주소를 받아 캐시에 저장한 후 브라우저에 전달합니다.

---

### Q. DNS 레코드 타입의 종류를 설명해주세요.

A 레코드는 도메인을 IPv4 주소로 변환하고, AAAA 레코드는 도메인을 IPv6 주소로 변환합니다. CNAME은 도메인에 별칭을 붙여 다른 도메인으로 연결하고, MX는 해당 도메인의 메일 서버 주소를 가리킵니다. NS는 도메인의 네임서버 주소를 나타내고, TXT는 도메인 인증이나 SPF 설정 같은 텍스트 정보를 담습니다. PTR은 IP 주소로 도메인을 역방향으로 조회할 때 사용합니다.

---

### Q. DNS는 왜 UDP를 사용하나요?

DNS는 기본적으로 UDP 포트 53번을 사용합니다.

DNS 조회와 응답은 매우 짧은 메시지이기 때문에 TCP 연결 수립의 오버헤드가 불필요합니다. 응답이 없을 경우 클라이언트가 재요청하면 되기 때문에 신뢰성 문제도 없습니다. 무엇보다 매 요청마다 3-way handshake를 하면 너무 느리기 때문에 빠른 응답이 중요한 DNS에는 UDP가 적합합니다.

단, 응답 크기가 512바이트를 초과하거나 영역 전송(Zone Transfer)이 필요한 경우에는 TCP를 사용합니다.

---

## 5. HTTP & HTTPS

### Q. HTTP란 무엇이고, 특징을 설명해주세요.

HyperText Transfer Protocol로, 클라이언트와 서버 사이에서 데이터를 주고받기 위한 응용 계층 프로토콜입니다.

주요 특징은 세 가지입니다. 첫째로 무상태(Stateless)인데, 서버는 클라이언트의 이전 요청 상태를 저장하지 않고 각 요청을 독립적으로 처리합니다. 상태를 유지하려면 쿠키나 세션, JWT 등을 사용합니다. 둘째로 비연결성(Connectionless)인데, 요청과 응답이 끝나면 연결을 끊습니다. HTTP/1.1의 Keep-Alive로 연결을 재사용하는 방식으로 개선되었습니다. 셋째로 클라이언트-서버 구조로 역할이 분리되어 있어 각각 독립적으로 발전할 수 있습니다.

---

### Q. HTTP 메서드의 종류와 특징을 설명해주세요.

GET은 리소스를 조회하는 메서드로, 멱등성과 안전성을 모두 가집니다. POST는 리소스를 생성하거나 데이터를 전송하는 메서드로, 멱등성과 안전성이 없습니다. PUT은 리소스 전체를 수정하거나 없으면 생성하는 메서드로, 멱등성은 있지만 안전하지 않습니다. PATCH는 리소스의 일부만 수정하는 메서드이고, DELETE는 리소스를 삭제하는 메서드로 멱등성이 있습니다.

멱등성은 같은 요청을 여러 번 보내도 결과가 동일한 성질이고, 안전성은 서버 상태를 변경하지 않는 성질입니다.

---

### Q. HTTP 상태 코드를 설명해주세요.

1xx는 요청을 받아 처리 중이라는 정보성 응답입니다. 2xx는 요청이 정상적으로 처리됐다는 성공 응답으로, 200 OK, 201 Created, 204 No Content가 대표적입니다. 3xx는 리다이렉션으로 추가 조치가 필요한 상태이고, 301은 영구 이동, 302는 임시 이동, 304는 캐시된 버전을 사용하라는 의미입니다. 4xx는 클라이언트 오류로, 400은 잘못된 요청, 401은 인증 필요, 403은 접근 금지, 404는 리소스 없음을 의미합니다. 5xx는 서버 오류로, 500은 서버 내부 오류, 503은 서버 일시 사용 불가를 의미합니다.

---

### Q. HTTP/1.1, HTTP/2, HTTP/3의 차이를 설명해주세요.

HTTP/1.0은 요청마다 TCP 연결을 새로 수립해서 매번 3-way handshake가 발생합니다.

HTTP/1.1은 Keep-Alive로 TCP 연결을 재사용하고 파이프라이닝을 도입해 여러 요청을 연속으로 보낼 수 있게 했습니다. 다만 앞 요청이 지연되면 뒤 요청도 대기하는 HOL Blocking 문제가 있습니다.

HTTP/2는 멀티플렉싱을 도입해 하나의 TCP 연결에서 여러 요청과 응답을 동시에 처리하여 HOL Blocking을 해결했습니다. 헤더 압축(HPACK)으로 전송량을 줄이고, 서버 푸시로 클라이언트 요청 없이 리소스를 미리 보낼 수 있습니다.

HTTP/3는 TCP 대신 UDP 기반의 QUIC 프로토콜을 사용해 TCP의 HOL Blocking 문제를 근본적으로 해결했습니다. 연결 수립 시간도 단축되고, 모바일 환경에서 IP가 변경되어도 연결을 유지할 수 있습니다.

---

### Q. HTTPS란 무엇이고, HTTP와 차이점은 무엇인가요?

HTTPS는 HTTP에 TLS(Transport Layer Security) 암호화를 더한 프로토콜입니다.

HTTP는 데이터를 평문으로 전송하지만, HTTPS는 데이터를 암호화해 도청과 변조를 방지합니다. 또한 CA가 발급한 인증서를 통해 서버의 신원을 보장하고, 전송 중 데이터가 변조되지 않았음을 무결성으로 보장합니다. HTTP는 기본 포트 80번을, HTTPS는 443번을 사용합니다.

---

### Q. HTTPS의 TLS 핸드셰이크 과정을 설명해주세요.

먼저 클라이언트가 지원하는 TLS 버전과 암호화 방식 목록, 랜덤값을 담은 Client Hello를 서버에 보냅니다. 서버는 암호화 방식을 선택하고 자신의 랜덤값을 담은 Server Hello와 함께 공개키가 담긴 인증서를 전송합니다.

클라이언트는 CA를 통해 인증서를 검증한 후, 서버의 공개키로 암호화한 Pre-master Secret을 전송합니다. 양측은 이 값과 각자의 랜덤값을 조합해 동일한 세션 키(대칭키)를 생성합니다. 이후 Finished 메시지를 교환하며 핸드셰이크를 마치고 HTTPS 통신을 시작합니다.

핵심은 비대칭키로 세션 키를 안전하게 교환하고, 실제 통신은 빠른 대칭키 암호화를 사용한다는 점입니다.

---

### Q. REST API란 무엇인가요?

Representational State Transfer로, HTTP를 기반으로 자원을 URI로 표현하고 HTTP 메서드로 행위를 정의하는 API 설계 아키텍처 스타일입니다.

REST는 6가지 원칙을 가집니다. 클라이언트와 서버의 역할을 분리하고, 각 요청에 필요한 모든 정보를 담아 무상태로 처리합니다. 응답에 캐싱 가능 여부를 명시하고, 중간에 프록시나 로드밸런서 같은 계층을 추가할 수 있습니다. URI로 자원을 식별하고 표준 HTTP 메서드를 사용하는 균일한 인터페이스를 유지합니다.

---

### Q. CORS(Cross-Origin Resource Sharing)란 무엇인가요?

웹 브라우저의 동일 출처 정책(Same-Origin Policy)에 의해 다른 출처의 리소스 요청이 기본적으로 차단되는데, 이를 허용하기 위한 HTTP 헤더 기반 메커니즘입니다.

출처는 프로토콜, 도메인, 포트 번호가 모두 같아야 동일 출처입니다. 브라우저가 다른 출처에 요청할 때 Origin 헤더를 포함해 전송하면, 서버가 응답에 Access-Control-Allow-Origin 헤더로 허용 여부를 알려줍니다. POST나 PUT 같은 복잡한 요청은 실제 요청 전에 OPTIONS 메서드로 Preflight 요청을 먼저 보내 서버가 허용하는지 확인합니다.

---

### Q. 웹 브라우저에 URL을 입력하면 일어나는 과정을 설명해주세요.

네트워크 면접에서 가장 자주 나오는 종합 질문입니다.

먼저 브라우저가 URL을 파싱해 프로토콜, 도메인, 경로 등을 분석합니다. 그다음 DNS 조회를 통해 도메인을 IP 주소로 변환합니다. IP 주소를 얻으면 해당 서버와 TCP 3-way handshake로 연결을 맺고, HTTPS라면 TLS 핸드셰이크로 암호화 세션도 수립합니다.

연결이 완료되면 HTTP 요청을 전송하고, 서버는 요청을 처리해 HTML 응답을 반환합니다. 브라우저는 HTML을 파싱해 DOM을 생성하고, CSS와 JS를 로드해 화면에 렌더링합니다. 마지막으로 TCP 연결을 4-way handshake로 종료하거나 Keep-Alive로 유지합니다.

---

## 6. 쿠키 & 세션 & JWT

### Q. 쿠키(Cookie)와 세션(Session)의 차이를 설명해주세요.

HTTP는 무상태 프로토콜이기 때문에 상태를 유지하려면 쿠키나 세션이 필요합니다.

쿠키는 데이터를 클라이언트(브라우저)에 저장합니다. 서버에 매번 조회할 필요가 없어 빠르지만, 클라이언트에 노출되어 탈취 위험이 있고 최대 4KB까지만 저장할 수 있습니다. 주로 자동 로그인이나 장바구니에 사용합니다.

세션은 데이터를 서버에 저장하고, 클라이언트에는 세션 ID만 전달합니다. 데이터가 서버에 있어 보안이 높고 용량 제한도 적지만, 서버 메모리를 사용하고 매 요청마다 서버에서 조회해야 해 속도가 느립니다. 주로 로그인 상태 유지에 사용합니다.

---

### Q. JWT(JSON Web Token)란 무엇인가요?

서버가 발급하는 자가 서명된 토큰입니다. 세션처럼 서버에 상태를 저장하지 않고 클라이언트에 인증 정보를 담아 무상태 인증을 구현합니다.

JWT는 점(.)으로 구분된 세 부분으로 구성됩니다. Header에는 토큰 타입과 서명 알고리즘 정보가, Payload에는 사용자 정보와 만료 시간 같은 클레임이, Signature에는 Header와 Payload를 비밀키로 서명한 값이 담깁니다.

장점은 서버가 세션 저장소 없이 토큰 서명만 검증하면 되므로 분산 서버 환경에 유리합니다. 단점은 토큰이 탈취되면 만료 전까지 무효화하기 어렵고, Payload는 Base64 인코딩이라 민감한 정보를 담으면 안 됩니다.

---

### Q. 세션 기반 인증과 토큰 기반 인증의 차이를 설명해주세요.

세션 기반 인증은 서버에 세션 정보를 저장합니다. 서버 메모리를 사용하고, 서버가 여러 대면 세션 공유 문제가 발생해 Redis 같은 별도 저장소가 필요합니다. 대신 세션 ID가 탈취되더라도 서버에서 바로 무효화할 수 있어 보안 관리가 용이합니다.

토큰 기반 인증(JWT)은 서버가 상태를 저장하지 않고 토큰 서명만 검증합니다. 서버를 수평 확장하기 쉽지만, 토큰 자체에 정보가 담겨 있어 탈취되면 만료 전까지 막기 어렵습니다. 이를 보완하기 위해 짧은 유효기간의 Access Token과 긴 유효기간의 Refresh Token을 함께 사용합니다.

---

## 7. 보안

### Q. 대칭키 암호화와 비대칭키 암호화의 차이를 설명해주세요.

대칭키 암호화는 암호화와 복호화에 같은 키를 사용합니다. 속도가 빠르지만 키를 상대방에게 안전하게 전달하는 키 배송 문제가 있습니다. AES, DES가 대표적인 알고리즘입니다.

비대칭키 암호화는 공개키로 암호화하고 개인키로 복호화합니다. 공개키는 누구에게나 공개해도 되므로 키 배송 문제가 없지만, 대칭키보다 속도가 느립니다. RSA, ECC가 대표적입니다.

TLS는 비대칭키로 세션 키를 안전하게 교환한 후, 실제 통신은 빠른 대칭키로 처리하는 하이브리드 방식을 사용합니다.

---

### Q. XSS(Cross-Site Scripting)와 CSRF(Cross-Site Request Forgery)의 차이를 설명해주세요.

XSS는 공격자가 웹 페이지에 악성 스크립트를 삽입해 다른 사용자의 브라우저에서 실행시키는 공격입니다. 예를 들어 게시글에 쿠키를 탈취하는 스크립트를 심어두면, 해당 페이지를 방문한 사용자의 쿠키가 공격자에게 전달됩니다. 입력값 검증과 인코딩, CSP 헤더 적용, HttpOnly 쿠키 설정으로 방어합니다.

CSRF는 인증된 사용자가 의도하지 않은 요청을 서버에 보내도록 유도하는 공격입니다. 로그인된 사용자가 악성 사이트를 방문하면 자동으로 송금 요청이 발생하는 식입니다. CSRF 토큰, SameSite 쿠키 속성, Referer 헤더 검증으로 방어합니다.

두 공격의 차이는 XSS는 피해자의 브라우저를 공격해 스크립트를 실행시키는 것이고, CSRF는 피해자의 인증 정보를 이용해 서버에 위조 요청을 보내는 것입니다.

---

### Q. SQL Injection이란 무엇이고, 어떻게 방어하나요?

악의적인 SQL 구문을 입력값에 삽입해 데이터베이스를 무단으로 조작하는 공격입니다. 예를 들어 로그인 폼에 `' OR '1'='1`을 입력하면 WHERE 조건이 항상 참이 되어 모든 사용자 정보가 조회될 수 있습니다.

방어 방법으로는 SQL과 데이터를 분리해 처리하는 PreparedStatement나 파라미터 바인딩을 사용하는 것이 가장 중요합니다. ORM을 사용해 직접 SQL 작성을 줄이고, 입력값에서 특수문자를 필터링하며, DB 계정에 필요한 권한만 부여하는 최소 권한 원칙도 적용해야 합니다.

---

## 8. 기타 네트워크 개념

### Q. 로드 밸런싱(Load Balancing)이란 무엇인가요?

여러 서버에 네트워크 트래픽을 균등하게 분산시키는 기술입니다.

주요 알고리즘으로는 순서대로 순환하며 분배하는 라운드 로빈, 서버 성능에 따라 가중치를 부여하는 가중 라운드 로빈, 현재 연결 수가 가장 적은 서버에 보내는 최소 연결, 클라이언트 IP를 해시해 항상 같은 서버로 보내는 IP 해시 방식이 있습니다.

L4 로드 밸런서는 IP와 포트 기반으로 분산하고, L7 로드 밸런서는 HTTP URL이나 헤더 내용까지 보고 더 세밀하게 분산할 수 있습니다.

---

### Q. 프록시(Proxy)와 리버스 프록시(Reverse Proxy)의 차이를 설명해주세요.

프록시(포워드 프록시)는 클라이언트 앞에 위치해 클라이언트를 대신해서 서버에 요청합니다. 서버는 클라이언트의 실제 IP를 알 수 없고, 클라이언트의 익명성 보장, 접근 제어, 캐싱 등에 사용합니다.

리버스 프록시는 서버 앞에 위치해 클라이언트 요청을 대신 받아 내부 서버로 전달합니다. 클라이언트는 실제 서버를 알 수 없고, 로드밸런싱, 보안, 캐싱 등에 사용합니다. Nginx와 CDN이 대표적인 예입니다.

---

### Q. CDN(Content Delivery Network)이란 무엇인가요?

전 세계 여러 지역에 엣지 서버를 분산 배치해, 사용자와 가장 가까운 서버에서 콘텐츠를 제공하는 기술입니다.

물리적으로 가까운 서버에서 응답하므로 지연 시간이 줄어들고, 이미지나 CSS, JS 같은 정적 파일을 캐싱해 원본 서버의 부하를 줄입니다. 특정 서버에 장애가 발생해도 다른 엣지 서버가 대응할 수 있어 가용성도 높아집니다.

---

### Q. 웹소켓(WebSocket)이란 무엇이고, HTTP와 어떻게 다른가요?

클라이언트와 서버 간 양방향 실시간 통신을 지원하는 프로토콜입니다.

HTTP는 클라이언트가 요청해야만 서버가 응답하는 단방향 통신이고, 요청-응답 후 연결이 끊깁니다. 반면 WebSocket은 한 번 연결하면 클라이언트와 서버 모두 능동적으로 데이터를 보낼 수 있는 양방향 지속 연결입니다.

WebSocket 연결은 HTTP 업그레이드 요청(`Upgrade: websocket`)으로 시작하며, 채팅, 실시간 알림, 주식 시세 등에 사용합니다.

---

### Q. HTTP 폴링(Polling), 롱 폴링(Long Polling), SSE, 웹소켓의 차이를 설명해주세요.

모두 실시간 통신을 구현하는 방법들이지만 방식이 다릅니다.

폴링은 클라이언트가 주기적으로 서버에 요청을 보내는 방식으로, 구현이 단순하지만 데이터가 없어도 불필요한 요청이 계속 발생합니다.

롱 폴링은 클라이언트가 요청을 보내면 서버가 데이터가 생길 때까지 응답을 지연하다가 데이터가 생기면 응답하는 방식입니다. 실시간에 가깝지만 연결 유지 비용이 존재합니다.

SSE(Server-Sent Events)는 서버에서 클라이언트로만 데이터를 스트리밍하는 단방향 방식으로, HTTP를 그대로 사용할 수 있습니다.

웹소켓은 양방향 지속 연결로, 클라이언트와 서버 모두 자유롭게 데이터를 보낼 수 있어 실시간 양방향 통신에 가장 적합합니다.
