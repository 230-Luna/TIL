### Q. 컴퓨터는 운영체제가 있어야 동작하는가?

없어도 동작한다.
하지만 운영체제가 없으면 처음 설계한 그대로만 동작할 뿐 다른 기능을 추가할 수 없다.
예전에 사용하던 유선 전화기는 통화기능만 가능하고 별다른 기능이 없지만, 운영체제가 있는 스마트폰은 애플리케이션을 설치해 기능을 추가할 수 있다.

### 운영체제가 하는 일

1. 프로세스 관리
2. 메모리 관리
3. 하드웨어 관리
4. 파일 시스템 관리

### 시분할 시스템

컴퓨터의 CPU의 실행 시간을 짧은 단위(Time Slice)로 나누어 여러 사용자나 여러 프로그램이 번갈아 가며 자원을 사용하여,
동시에 작업이 처리되는 것처럼 보이게 하는 방식.

> 사용자는 동시에 작업하는 것처럼 느낌.

### 커널

프로세서와 메모리, 저장장치를 관하는 핵심적인 기능을 담당
사용자는 운영체제의 커널에 직접 접근할 수 없다.
CLI, GUI와 같은 인터페이스를 통해 접근이 가능

### 시스템콜

어플리케이션은 시스템콜을 통해서 커널에 접근할 수 있다.

사용자나 어플리케이션이 하드디스크에 데이터를 저장할 때 직접 접근하면 중요한 데이터를 덮어쓸 수도 있다.
시스템콜을 이용하면 커널에서 제공하는 write 함수를 쓰게 되는데 그럼 운영체제가 알아서 하드디스크의 빈 공간에 저장하게 된다.

### 드라이버

하드웨어와 커널의 인터페이스로는 드라이버를 사용한다.
_운영체제는 많은 종류의 하드웨어를 전부 지원해야 하기 때문에 각각의 하드웨어에 맞는 프로그램을 커널이 미리 가지고 있기는 힘들다._
그래서 하드웨어를 제작한 제조사에서 드라이버를 만들어 제공하는 게 일반적이다.
보통 키보드나 마우스 같은 간단한 장치들은 커널에 포함되어 있어 컴퓨터에 꽂으면 바로 동작하지만 그래픽 카드나 타블렛 같은 복잡한 장치들은 디바이스 드라이버를 설치해서 사용해야한다.

### 프로그램 내장방식

*프로그램은 메모리에 올려서 실행*되는데 프로그램을 메모리에 내장했다고 해서 프로그램 내장방식이라고 한다.

## 컴퓨터의 하드웨어

### 메인보드

다른 하드웨어를 연결하는 장치.
장치간의 데이터를 전송하는 건 메인보드의 버스가 담당함.

CPU, 메모리, 하드디스크, 그래픽 카드 등 각 연결단자에 꽂아준다.
출력단자에는 모니터, USB단자에는 마우스와 키보드를 꽂아주면 된다.

### CPU 구조

중앙처리장치

CPU 구성요소

- 산술논리연산장치(ALU): CPU에서 실제로 데이터 연산을 담당
- 제어장치: 모든 장치들의 동작을 지시하고 제어하는 장치
- 레지스터: CPU 내에서 계산을 위해 데이터를 임시로 보관하는 장치(캐시)

### 메모리 종류

- RAM: 랜덤으로 데이터를 읽어도 저장된 위치와 상관없이 읽기 속도가 같음. 전력이 끊기면 데이터를 잃어버리기 때문에 주로 프로그램을 실행시키는 용도로 사용
- ROM: 전력이 끊겨도 데이터를 계속 보관할 수 있지만 데이터를 한번 쓰면 수정이 불가능. 컴퓨터 부팅과 관련된 BIOS를 저장하는데 주로 사용

### 컴퓨터 부팅과정

1. 사용자가 컴퓨터 전원을 누른다.
2. ROM에 있는 BIOS가 실행된다.
3. BIOS는 전원,CPU,메모리,키보드,마우스,하드디스크 등 주요 하드웨어에 이상이 없는지 체크한다.
4. 만약 주요 장치에 이상이 있다면 오류음(Beep Code)을 내면서 부팅이 이루어지지 않고, 이상이 없다면 하드디스크에 있는 마스터 부트 레코드에 저장된 부트로더를 메모리로 가져와서 실행한다.
   만약 윈도우즈 운영체제와 리눅스 운영체제가 둘 다 설치되어 있는 컴퓨터라면 어떤 운영체제를 실행할지 선택하는 화면이 나온다.
5. 운영체제를 선택했거나 운영체제가 하나면 바로 운영체제를 메모리로 불러오고 모니터에 바탕화면이 보이게 된다.
6. 이후로 실행되는 모든 응용프로그램은 메모리에 올라와서 운영체제가 관리한다.

### 폴링 방식

CPU가 입출력 장치에 데이터를 읽거나 쓰려고 하는 상황이라고 가정.
CPU는 입출력 작업이 들어오면 입출력 관리자에게 입출력 명령을 내린다.
CPU 관점에선 입출력 명령이 언제 완료될지 알 수 없기 때문에 주기적으로 계속 확인해줘야 한다.

폴링방식의 단점은 주기적으로 CPU가 확인해줘야 하니 성능이 좋지 않다는 것이다.

### 인터럽트

관리자는 입출력이 완료됐을 때 CPU에게 신호를 주고 CPU는 그 신호를 받아 인터럽트 서비스 루틴을 실행시켜 작업을 완료.
인터럽트 서비스 루틴은 특정 인터럽트가 들어오면 그 인터럽트를 처리하는 함수.

인터럽트는 비동기적으로 동작하기 때문에 성능에 이점이 있다.

인터럽트는 하드웨어 방식과 소프트웨어 방식 2개가 있다.
하드웨어 방식은 조금 전 설명한 입출력 등과 같은 인터럽트가 있다. 소프트웨어 방식은 사용자 프로그램에서 발생한 interrupt가 있다.
예를 들면 유효하지 않은 메모리에 접근하거나 0으로 나누는 명령어 등이 있음

### 프로그램

프로그램은 하드 디스크 등과 같은 저장장치에 저장된 명령문의 집합체
애플리케이션이나 앱이라고도 불리고 윈도우 운영체제에서는 .exe 파일의 모습을 하고 있음.

### 프로세스

실행중인 프로그램
= 하드디스크에 저장된 프로그램이 메모리에 올라갔을 때 실행중인 프로그램

프로그램은 컴퓨터 관점에서 하드디스크 즉 저장장치만 사용하는 수동적인 존재.
반면 프로세스는 메모리도 사용하고 운영체제의 CPU 스케줄링 알고리즘에 따라서 CPU도 사용하고 필요에 따라 입력과 출력을 하기 때문에 능동적인 존재.

### 프로세스의 구조

- 코드 영역: 자신을 실행하는 코드가 저장되어 있음.
- 데이터 영역: 전역 변수와 static 변수가 저장되어 있음.
- 스택 영역: 지역 변수와 함수를 호출했을 때 필요한 정보들이 저장됨.
- 힙 영역: 프로그래머가 동적으로 메모리를 할당하는 데 쓰임

### 컴파일 과정

C언어 기준(두 숫자를 더하는 코드)

1. 전처리를 거쳐서 매크로로 정의한 숫자를 치환하고 필요한 파일을 불러온다.
2. 전처리기를 거치면 파일의 확장자는 .i 파일이 된다
3. 그 다음 컴파일러가 컴파일을 해준다.
4. 컴파일을 마치면 고수준인 C언어를 저수준인 어셈블리어로 바꿔준다. 어셈블리어는 명령어가 기계어랑 1대1 매칭이 되기 때문에 기계어랑 가장 가까운 언어이다.
5. 컴파일러를 거치면 파일의 확장자는 .s가 된다.
6. 이제 어셈블러가 어셈블리어를 기계어로 바꿔준다.
7. 그럼 이 파일은 0과 1로 이루어진 기계어로 구성되고 파일의 확장자는 .o가 된다. 기계어로 구성되어 있기 때문에 파일을 텍스트 에디터로 열어보면 글씨가 깨져서 보인다.
8. 마지막으로 링커가 링킹을 한다. 여러가지 라이브러리나 다른 소스 코드를 연결한다.

이렇게 링킹까지 거치면 파일의 확장자는 .exe가 된다.

### 프로그램 실행과정

1. 위에서 만들어진 .exe파일을 더블클릭하면 하드디스크에 있는 test.exe 파일이 메모리에 올라가게 되고 이렇게 올라간 프로그램은 프로세스라는 새로운 이름으로 불리게 된다.
   이 프로세스는 이제부터 운영체제에 의해 관리된다.
2. CPU 내 제어 장치가 두 숫자를 메모리에 저장시킨다.
3. 이 메모리에 저장된 값을 EDX,EAX 레지스터로 가져온다.
4. 제어장치가 레지스터에 저장된 두 숫자를 가지고 더하라는 명령을 한다.
5. 산술 논리연산장치가 두 숫자를 더하고 그 결과는 EAX 레지스터에 저장한다.
6. 다시 제어장치가 EAX 레지스터에 저장된 숫자(결과값)를 가져와서 메모리에 저장시킨다다.

### 유니 프로그래밍

메모리에 오직 하나의 프로세스가 올라와 처리하는 것.
CPU는 프로세스를 실행하다가 I/O 작업을 만나면 I/O 작업이 완료될 때까지 기다리게 된다. 그러다가 I/O 작업이 완료되면 작업을 계속 이어서 처리할 수 있다.
이러한 문제 때문에 유니프로그래밍은 프로세스가 IO 작업을 하는 동안 CPU가 쉬는 시간이 발생하는 단점이 있고,
하나의 프로세스를 다 끝내야 다른 프로세스를 실행할 수 있다는 단점이 있다.

### 멀티프로그래밍

메모리에 여러개의 프로세스를 올려서 이들을 처리함.
멀티프로그래밍은 CPU가 프로세스를 실행하다가 I/O 작업을 만나면 해당 프로세스에서 발생한 I/O 작업은 기다리면서 다른 프로세스를 실행한다. 그러면 CPU가 쉬는 시간이 적어져 더 효율성이 높아지게 된다.

### 멀티태스킹

여기서 프로세스가 I/O 작업이 있을 때만 다른 프로세스를 실행하는 방식이 아니라 한 실행하면서 모든 프로세스를 동시에 실행시키는 것처럼 느끼게 하는 기술

### 멀티프로세서

CPU를 한개가 아니라 여러개를 이용할 수 있는데 CPU가 여러개 있는 것을 말함.

멀티프로세서로 작업을 처리하는 것을 멀티프로세싱이라고 부른다.

### PCB(Process Control Block)

프로세스의 모든 정보를 저장하는 자료구조
CPU에 의해 실행 중인 특정한 프로세스를 관리할 필요가 있는 정보를 포함하는 운영체제 커널의 자료 구조.

프로세스가 만들어지면 운영체제는 해당 프로세스의 정보를 가지고 있는 PCB를 만들고 저장한다.

PCB들은 연결리스트라는 자료구조로 저장한다.

```
PCB A -> PCB B -> PCB C
```

운영체제는 프로세스가 종료되면 연결 리스트에서 해당 프로세스의 PCB를 제거한다.
연결 리스트 구조는 유지된다.

```
PCB A ->  PCB C
```

### PCB 구성요소

- 프로세스 상태(Process State)
- 프로세스 식별자(PID)
- 프로그램 카운터(Program Counter): 다음에 실행할 명령어의 주소
- CPU 레지스터 (CPU Registers)
- CPU 스케줄링 정보
- 메모리 관리 정보
- 입출력 상태 정보

### 프로세스 상태

- 생성 상태
  PCB를 생성하고 메모리에 프로그램 적재를 요청. 메모리에 프로그램 적재를 승인받으면 준비 상태로 넘어감.
- 준비 상태
  CPU를 사용하기 위해 기다리고 있는 상태. 준비 상태에 있는 프로세서는 이후에 배울 CPU 스케줄러에 의해 CPU가 할당된다. 대부분의 프로세스는 준비 상태에 있다.
- 실행 상태
  CPU 스케줄러에 의해 CPU를 할당받아 실행되는 상태. 실행 상태에 있는 프로세스의 수는 CPU의 개수만큼이다. CPU가 1개라면 실행 상태의 프로세스는 최대 1개임.
  실행 상태에 있는 프로세스도 CPU를 무한정 쓸 수 있는 것이 아니라 부여된 시간만큼만 사용할 수 있다.
  CPU 스케줄러는 부여된 시간을 초과하면 할당된 CPU를 강제로 빼앗는다. 그럼 프로세스는 다시 준비 상태로 돌아간다.
- 대기 상태
  프로세스가 입출력 요청을 하면 입출력 완료될 때까지 기다리는 상태.
  CPU는 굉장히 빠른 장치인 반면 입출력 작업은 상당히 느리다.
  특정 프로세스가 입출력 요청을 하면 입출력이 완료될 때까지 CPU를 기다리게 하는 건 굉장히 비효율적이다.
  대신 입출력 요청을 한 프로세스를 대기 상태로 두고 다른 프로세스에게 CPU를 할당하다가 입출력 작업이 완료되면 대기 상태에 있던 프로세스에게 CPU 할당 기회를 준다.
  이렇게 하면 CPU에게 쉬는 시간을 주지않고 일을 시킬 수 있다.
- 완료 상태
  프로세스가 종료된 상태
  프로세스가 사용했던 데이터를 메모리에서 제거하고 생성된 PCB도 제거

### 프로세스 생성과 종료

컴퓨터가 부팅되면 운영체제는 최초의 프로세스 하나를 생성한다.
이 프로세스는 0번이라는 식별자가 부여되어 0번 프로세스라고 불린다.
운영체제에 필요한 다른 프로세스들은 새로 생성하는 대신 이 0번 프로세스를 복사하여 생성한다.

일반 애플리케이션 역시 자신이 실행되는 계층의 부모 프로세스를 복사해 실행된다.
이렇게 프로세스를 복사하는 방식을 사용하는 이유는 새로 생성하는 것보다 더 저렴하기떄문이다.

이때 복사된 프로세스를 자식 프로세스라고 하며 자식 프로세스 입장에서 자신을 복사한 프로세스는 부모 프로세스가 된다.
자식 프로세스는 부모 프로세스의 코드 영역, 데이터 영역, 스택 영역과 PCB의 내용을 모두 복사한다.

fork 함수로 프로세스를 복사한 후 exec 함수를 실행하면 복사된 자식 프로세스의 코드와 데이터 영역이 원하는 값으로 덮어쓰여진다.

### 좀비 프로세스

실행이 종료되었지만 부모 프로세스가 종료 상태를 수거하지 않아 프로세스 테이블에 PCB가 남아 있는 프로세스

- 발생과정

1. 부모 프로세스가 자식 프로세스를 생성 (fork())
2. 자식 프로세스 실행 후 종료 (exit())
3. 부모 프로세스가 자식 프로세스의 종료 상태를 읽지 않음
4. 자식 프로세스의 PCB가 프로세스 테이블에 남아 좀비 프로세스가 됨

> exit 함수는 자식 프로세스가 부모 프로세스에게 종료를 알리는 함수임
> 부모 프로세스는 자식 프로세스의 exit status를 읽고 자식 프로세스를 정리함
> 자식 프로세스가 exit 신호를 보내지 못한 채 비정상으로 종료되거나 부모 프로세스가 자식 프로세스보다 먼저 종료되어 exit status를 읽지 못하면 해당 자식 프로세스는 메모리에 계속 남아 있게 된다.

컴퓨터를 오래 켜두면 느려지는 현상이 발생하는데,
이는 메모리에 많은 프로세스가 누적된 것도 있지만 좀비 프로세스(zombie process)가 쌓여 메모리를 차지해 느려지기도 한다.
컴퓨터 재부팅하면 메모리가 초기화되어 다시 빨라진다.

### 쓰레드

프로세스의 수가 많아지면 프로세스의 수만큼 PCB,코드,데이터,스택,힙 영역도 만들어줘야 하기 때문에 너무 무거워진다.
예를 들어, 웹브라우저를 실행시키면 프로세스가 1개 생성되고 탭을 1개 추가하면 기존 프로세스를 복사해서 총 2개의 프로세스가 존재하게 된다. 만약 탭 20개를 만드면 웹브라우저가 메모리를 너무 많이 차지한다.
이 때 쓰레드가 고안됐다.

프로세스 내에 존재하는 것으로 1개 이상 있을 수 있다.
한 프로세스 내의 쓰레드들은 그 프로세스의 PCB, 코드, 데이터, 힙 영역을 공유한다.
스택은 공유하지 않고 쓰레드마다 하나씩 가지고 있다.
TCB를 통해 쓰레드를 관리할 수 있다.

### 프로세스와 쓰레드의 장단점

1. 안정성
   프로세스는 서로 독립적이기 때문에 하나의 프로세스가 문제가 있더라도 다른 프로세스가 영향을 받지 않는다.
   반면 쓰레드는 하나의 프로세스 내에 존재하기 때문에 해당 프로세스에 문제가 생기면 그 안에 있는 모든 쓰레드에 문제가 생기게 된다.

   이러한 이유로 안정성 측면에서는 프로세스 방식이 쓰레드 방식보다 더 우수하다.

2. 속도와 자원
   각각의 프로세스는 서로 고유한 자원을 가지고 있다.

   코드,데이터,힙,스탭 영역을 전부 따로 두고 있고 프로세스 간의 통신을 하려면 IPC를 이용해야 해서 오버헤드가 크고 속도가 느리다.

   반면 쓰레드는 한 프로세스 내에서 스택 영역을 제외한 영역은 모두 공유하기 때문에 오버헤드가 굉장히 적다.

## CPU 스케줄링

프로그램을 실행시키면 메모리에 프로세스가 생성되고 각 프로세스에는 한 개 이상의 쓰레드가 있다.
프로세스들은 CPU를 차지하기 위해서 운영체제의 명령을 기다리고 있는다.
운영체제는 모든 프로세스에게 CPU를 할당/해제하는데 이를 CPU 스케줄링이라고 한다.

- 고려사항
  어떤 프로세스에게 CPU 리소스를 줘야 하는가?
  CPU를 할당받은 프로세스가 얼마의 시간동안 CPU를 사용해야 하는가?

위 두가지 고려사항이 컴퓨터의 성능에 굉장히 큰 영향을 미친다.
CPU를 할당받아 실행하는 작업을 CPU 버스트라고 부르고,
입출력 작업을 I/O burst라고 부른다.

### 스케줄링 목표

1. 리소스 사용률
2. 오버헤드 최소화
3. 공평성
4. 처리량
5. 대기시간
6. 응답시간

### 평균 대기시간

프로세스 여러 개가 실행될 때 이 프로세스들 모두가 실행되기까지 대기시간의 평균

### FIFO

First In First Out.
먼저 들어온 작업이 먼저 나간다는 뜻으로 스케줄링 큐에 들어온 순서대로 CPU를 할당받는 방식
FIFO 알고리즘은 프로세스의 Burst Time에 따라 성능의 차이가 심하게 나기 때문에 현대 운영체제에서 잘 쓰이지 않고 일괄 처리 시스템에 쓰인다.

### 콘보이 효과

Convoy Effect
실행 시간이 긴 프로세스 하나 때문에 여러 짧은 프로세스들이 줄줄이 기다리는 현상

### SJF

Shortest Job First
Burst Time이 짧은 프로세스를 먼저 실행하는 알고리즘

### SJF 구현 문제점

1. 어떤 프로세스가 얼마나 실행될지 예측하기 힘들다
   사용자가 브라우저 실행시키고 뮤직플레이어 실행시켰을 때 언제 종료할지 예측 불가능
2. Burst Time이 긴 프로세스는 아주 오랫동안 실행되지 않을 수 있다
   Burst Time이 짧은 프로세스가 중간에 계속 들어오면 다 종료될 때까지 기다려야함

### 타임 슬라이스, 타임 퀀텀

프로세스에게 할당하는 일정 시간

### RR

Round Robin

FIFO 알고리즘에서 단점을 해결.
한 프로세스에게 일정 시간만큼 CPU를 할당하고,
할당된 시간이 지나면 강제로 다른 프로세스에게 일정 시간만큼 CPU를 할당
강제로 CPU를 뺏긴 프로세스는 큐의 가장 뒷 부분으로 밀려남.

평균 대기 시간이 비슷하다면 라운드 로빈 알고리즘이 더 비효율적인 방식이다.
라운드 로빈 알고리즘은 컨텍스트 스위칭이 있기 때문에 컨텍스트 스위칭 시간이 더 추가되기 떄문이다.

라운드 로빈 알고리즘의 성능은 타임 슬라이스 값에 따라 크게 달라진다.

### 오버헤드

타임슬라이스를 너무 작게 설정해버리면 컨텍스트 스위칭이 너무 자주 일어나게 되고 타임 슬라이스에서 실행되는 프로세스의 처리량보다 컨텍스트 스위칭을 처리하는 양이 훨씬 커져서 배보다 배꼽이 더 커지는 상황이 발생한다. 이러한 상황을 '오버헤드가 크다'라고 한다.

### MLFQ

Multi Level Feedback Queue
RR의 업그레이드된 알고리즘

여러 개의 우선순위 큐를 사용해서 프로세스의 우선순위를 동적으로 바꾸는 CPU 스케줄링 알고리즘

우선순위를 가진 큐를 여러개 준비한다.
우선순위가 높으면 타임 슬라이스가 작고, 우선순위가 낮을수록 타임 슬라이스 크기가 커진다.
타임 슬라이스 크기를 오버해서 강제로 CPU를 뺏긴다면 원래 있던 큐보다 우선순위가 더 낮은 큐로 이동하게 된다.
그럼 다음번에 실행될 때는 타임 슬라이스가 조금 더 커지게 되고 여기서도 부족하면 다음엔 더 큰 타임 슬라이스를 할당받게 된다.

## 프로세스 동기화

### IPC

프로세스들 사이에서 데이터를 주고받거나 서로 동기화하기 위한 메커니즘

### IPC 방식

#### 메시지 전달

메시지 전달 방식은 한 프로세스가 다른 프로세스와 메모리를 공유하지 않고 데이터를 전달하는 방식.

전달 방식 분류

- 파이프
  운영체제가 생성한 통로를 통해 한 프로세스가 다른 프로세스에게 데이터를 전송하거나 읽는 방식.
  파이프는 스트림 형태의 단방향 통신이므로 송신 프로세스와 수신 프로세스가 반드시 동시에 실행되어야 함.
- 메시지 큐
  송신 프로세스가 보낸 데이터를 임시 저장소인 큐에 보관했다가 수신 프로세스가 필요할 때 가져가는 방식. 두 프로세스가 동시에 실행될 필요가 없으며 양방향 통신이 가능
- 소켓 통신
  메시지 큐보다는 복잡하지만 같은 컴퓨터뿐만 아니라 다른 네트워크의 프로세스와도 통신할 수 있다. 또한 양방향 통신과 비동기 통신이 모두 가능한 유연한 방식이다.
  클라이언트 서버 구조로

#### 공유 메모리 방식

공유 메모리 방식은 프로세스가 메모리를 공유하여 데이터를 직접 읽고 쓰면서 통신하는 방식.
공유 메모리는 크게 세마포어, 뮤텍스, 모니터 등이 있다.

이 방식은 메모리를 공유하는 과정에서 치명적인 버그가 발생할 수 있는데 이러한 문제를 동기화 문제라고 한다.

동기화 문제는 공유 메모리뿐만 아니라 동일한 프로세스 내에서 메모리를 공유하는 쓰레드에서도 발생할 수 있다.

### 임계 구역

여러 프로세스가 동시에 사용하면 안되는 영역

### 경쟁 조건

공유자원을 서로 사용하기 위해 경쟁하는 것

### 상호 배제

동시성 프로그래밍에서 여러 프로세스나 스레드가 공유 자원에 동시에 접근할 때 발생하는 데이터 경쟁(Race Condition)을 막기 위해, 한 번에 단 하나만 자원을 사용하도록 보장하는 동기화 기법

### 상호 배제 메커니즘 요구사항

1.  주어진 시간에 항상 하나의 프로세스만 임계구역에 접근할 수 있어야 한다.
2.  동시에 여러 개의 요청이 있더라도 하나의 프로세스만 진입하도록 허용해야 한다.
3.  임계구역에 들어간 프로세스는 최대한 빠르게 나와야 한다.

### 세마포어

여러 프로세스나 스레드가 공유 자원에 동시에 접근하지 못하도록 제어하는 동기화 도구
공유 자원 접근 개수를 관리하는 정수형 변수

단점

- 프로그래밍 오류 가능성(Signal/wait 실수)
- Deadlock 발생 가능.
- 우선순위 전도
- 디버깅 어려움

### 모니터

세마포어의 단점을 해결한 상호 배제 메커니즘.
세마포어 사용 시 발생할 수 있는 오류 가능성을 줄이고, 프로그래밍 언어 수준에서 동기화 기능을 캡슐화하여 제공하는 메커니즘
언어 차원에서 뮤텍스, 세마포어 등을 기반으로 구축한 고수준 추상화

### 경합 조건

여러 프로세스가 동시에 공유 자원을 수정하려 할 때 접근 순서에 따라 결과가 달라지는 것

### 데드락

교착상태
두 개 이상의 프로세스가 서로가 가진 자원을 기다리면서 영원히 실행되지 못하는 상태

교착상태 필요조건

1. 상호배제: 한 번에 하나의 프로세스만 자원 사용
2. 비선점: 운영체제가 강제로 자원을 뺏을 수 없음
3. 점유와 대기: 자원을 가진 상태에서 다른 자원을 기다림
4. 원형 대기: 프로세스들이 원형으로 서로 기다림

## 중간정리

오늘날 컴퓨터는 폰 노이만 구조를 하고 있다.
폰 노이만 구조의 필수 장치는 CPU와 메모리이다.
그리고 컴퓨터의 성능이나 사용자의 편의성을 위한 보조장치가 있다.

컴퓨터가 부팅되면 보조 장치인 하드 디스크에 저장되어 있는 운영체제가 메모리에 올라온다.
모든 프로그램은 운영체제에 의해 관리된다.
프로그램도 하드디스크에 저장되어 있는데, 사용자가 해당 프로그램을 실행하면 메모리로 올라간다.

메모리에 올라간 프로그램을 프로세스라고 부른다.
프로세스는 자기를 실행 시켜줄 CPU를 기다린다.
운영체제가 중간에서 CPU를 할당해주는 것을 CPU 스케줄링이라고 부른다.

CPU 스케줄링은 공평함과 성능 문제 때문에 프로세스들에게 일정시간, 즉 타임슬라이스만큼 CPU를 할당하기로 했고 이 때문에 동기화 문제가 발생한다.
이를 해결하기 위한 방법으로 세마포어와 모니터가 있다.

동기화 문제를 해결하기 위해 공유된 자원을 한 프로세스가 점유하게 만들었는데 교착상태가 발생했다. 교착상태가 발생하기 위해서는 상호배제, 비선점, 점유와대기, 원형대기가 일어나야한다.

## 메모리

[속도 빠름, 용량 작음, 가격 비쌈]
레지스터 > 캐시 > 메인메모리(RAM) > 보조저장장치(HDD,SSD)
[속도 느림, 용량 큼, 가격 쌈]

### 경계 레지스터

운영체제에서 사용자 프로세스가 할당된 메모리 영역을 벗어나 다른 프로그램이나 OS 영역을 침범하지 못하도록, 접근 가능한 메모리의 최대 주소값(한계치)을 저장하여 메모리를 보호하는 하드웨어 레지스터

### 절대 주소(물리 주소)

- 실제 메모리(RAM)에 존재하는 주소

### 상대 주소(논리 주소)

사용자, 프로세스, CPU

- CPU가 생성하는 주소
- 프로그램 기준 주소
- 실제 메모리 위치가 아님

### 주소 변환 과정

논리 주소는 CPU가 생성하는 프로그램 기준 주소이며
MMU를 통해 실제 메모리의 물리 주소로 변환된다.

### 메모리 할당 방식

연속 메모리 할당: 고정분할 메모리 할당, 가변 분할 메모리 할당

- 고정분할 메모리 할당
  메모리를 고정된 크기로 분할하여 각 영역에 하나의 프로세스를 할당하는 방식
  구조가 단순하여 구현이 쉽다.
  하지만 프로세스의 크기가 분할된 크기보다 작으면 공간이 낭비되는 내부단편화가 발생하고,
  프로세스의 크기가 분할된 크기보다 크면 실행이 불가능하다는 단점이 있다.
- 가변 분할 메모리 할당
  메모리를 프로세스의 크기에 맞춰 유동적으로 분할하여 할당하는 방식
  프로세스 크기에 맞게 메모리를 분할하므로 내부 단편화는 발생하지 않지만 외부 단편화가 발생할 수 있다.

### 외부 단편화

남은 메모리 공간의 총합은 충분하지만 이들이 연속되지 않아 프로세스를 실행할 수 없는 상황을 말한다.

## 가상 메모리

실제 RAM이 부족할 때 SSD나 하드 디스크의 일부를 메모리처럼 활용하여, 물리적 한계를 넘어 더 큰 프로그램 실행과 멀티태스킹을 가능하게 하는 운영체제의 메모리 관리 기술

가상 메모리 시스템에서 가상 주소는 메모리나 스왑 영역 한 곳에 위치한다.
메모리 관리자는 가상 주소와 물리 주소를 1대1 매핑 테이블로 관리한다.

### 동적 주소 변환(DAT)

가상 기억 시스템에서 프로그램 실행 시 CPU가 생성하는 가상 주소(논리 주소)를 실제 주기억 장치의 물리 주소로 실시간 변환하는 기술
MMU(Memory Management Unit) 같은 하드웨어가 주도하며, 동적 재배치를 통해 프로세스를 물리 메모리 내 다른 위치로 이동할 수 있게 하여 효율적인 메모리 관리를 지원

메모리 관리자는 물리 메모리를 어떻게 나눌지, 프로세스를 어디다 배치할지, 부족한 물리 메모리는 어떻게 처리할지와 같은 문제를 처리해야 하기 때문에 복잡한 과정을 거친다.
실제 물리 주소 0번지는 운영체제 영역이므로 프로세스가 사용할 수 없다.

가상 메모리 시스템은 물리 메모리 내용의 일부를 하드디스크에 있는 스왑 영역으로 옮기고 처리가 필요할 때 물리 메모리로 가져와 실행시키기 때문에 운영체제와 프로세스를 전부 실행시킬 수 있다.

물리 메모리와 스왑 영역을 합쳐서 프로세스가 사용하는 가상 주소를 물리 주소로 변환하는데 이것을 동적 주소 변환(Dynamic Address Translation, DAT)이라고 부른다.

### 가상 메모리 할당방식

가상 메모리 시스템에서는 운영체제 영역을 제외한 나머지 영역을 일정한 크기로 나누어서 프로세스에게 할당하는데 할당하는 방식은 메모리 분할 방식과 마찬가지로 가변 분할 방식과 고정 분할 방식이 있다.

- 가변 분할 방식: 세그멘테이션
- 고정 분할 방식: 페이징

세그멘테이션에는 외부단편화와 같은 단점이 있고 페이징에는 내부단편화와 같은 단점이 있기 때문에 이 단점을 보완한 세그멘테이션 페이징 혼용기법을 사용한다.
때문에 이 단점을 보완한 세그멘테이션 페이징 혼용기법을 사용한다.

### 세그멘테이션

세그멘테이션에서 프로그램은 함수나 모듈 등으로 세그먼트를 구성한다.
프로그램 입장에서 메모리를 살펴보면 메인 코드가 있는 세그먼트, 전역 데이터들이 있는 세그먼트, 힙 영역이 있는 세그먼트, 스택 영역이 있는 세그먼트 등이 있다.
각 세그먼트들은 메모리 영역에서 서로 인접할 필요는 없다.

반면 프로세스 입장에서는 우리가 앞서 봤던 것처럼 코드 영역,데이터 영역,힙 영역,스택 영역을 서로 인접한 것처럼 바라본다.

메모리 관리자는 세그멘테이션 테이블이라는 것을 가지고 있다.
세그멘테이션 테이블에는 Base Address와 Bound Address 정보가 저장되고 이걸 이용해 물리 메모리 주소를 계산한다.

Bound Address는 세그먼트의 크기를 나타낸다.
메모리 관리자는 CPU에서 받은 논리 주소와 이 Bound Address의 크기를 비교한다.
만약 논리주소가 Bound Address보다 작다면 논리주소와 Base Address를 더해 물리주소를 구하고 논리주소가 Bound Address보다 크다면 메모리를 침범했다고 생각하고 에러를 발생시킨다.

### 페이징

세그멘테이션 기법은 외부단편화 문제가 있기 때문에 이를 해결하기 위해 고안되었다.
